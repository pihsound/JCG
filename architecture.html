💡 핵심 원칙

좋은 모듈 설계 = 결합도 낮게(약하게) + 응집도 높게(강하게)
"각 모듈은 자기 일만 하고(응집도↑), 다른 모듈과 최소한으로 연결(결합도↓)"

01
소프트웨어 아키텍처 패턴
23년 1회24년 3회

소프트웨어 시스템의 전체적인 구조를 설계하는 재사용 가능한 해법. 반복적으로 발생하는 설계 문제에 대한 검증된 설계 방식.

🍰
레이어드 (계층형)
Layered Pattern
시스템을 계층으로 분리. 각 계층은 바로 아래 계층에만 의존. 상위 계층은 서비스 제공, 하위 계층은 서비스 사용.
예) OSI 7계층, MVC, 3-tier 아키텍처
🎮
클라이언트-서버
Client-Server Pattern
서비스를 요청하는 클라이언트와 서비스를 제공하는 서버로 분리. 서버는 여러 클라이언트 요청을 처리.
예) 웹 애플리케이션, 이메일 시스템
🚰
파이프-필터
Pipe-Filter Pattern
데이터를 파이프(연결)로 전달하고, 각 필터(처리 단계)가 변환. 처리 단계를 독립적으로 교체 가능.
예) Unix 쉘 명령어, 컴파일러 (어휘→구문→의미 분석)
📢
이벤트-버스
Event-Bus Pattern
이벤트 소스가 이벤트 버스에 발행하면, 구독한 리스너가 처리. 느슨한 결합.
예) 알림 시스템, Android 이벤트 처리
🗂️
MVC
Model-View-Controller
Model(데이터) + View(화면) + Controller(로직)으로 분리. UI와 비즈니스 로직 분리.
예) Spring MVC, Django, Ruby on Rails
⭐
마스터-슬레이브
Master-Slave Pattern
마스터가 작업을 분배하고 슬레이브가 처리 후 결과 반환. 병렬 처리와 오류 허용에 유용.
예) DB 복제, 분산 컴퓨팅
02
결합도 (Coupling) — 낮을수록 Good
20년 1회21년 3회 24년 2회25년 1회

모듈과 모듈 간의 상호 의존 정도. 결합도가 낮을수록 모듈의 독립성이 높아지고 유지보수가 쉬워진다.

결합도 스펙트럼
왼쪽(낮음, 좋음) ──────────────── 오른쪽(높음, 나쁨)
👍 낮음(독립적) 👎 높음(의존적)
자료
Data
단순 데이터 파라미터 전달
스탬프
Stamp
배열·구조체 등 복합 자료구조 전달
제어
Control
제어 신호(flag)를 전달하여 흐름 제어
외부
External
외부 환경·형식·프로토콜에 의존
공통
Common
전역 변수(Global Variable) 공유
내용
Content
다른 모듈의 내부 변수·기능에 직접 접근
💡 암기법 — 결합도 낮음→높음

자스제외공내 — 자료 / 스탬프 / 제어 / 외부 / 공통 / 내용
"자스제외? 공내!!" — 자료부터 시작해서 내용이 가장 나쁜 것!

유형설명기출 포인트
자료 결합도모듈 간에 단순 자료형(정수, 문자 등) 파라미터로만 데이터 전달가장 낮음, 이상적
스탬프 결합도배열·구조체·객체 같은 복합 자료구조를 파라미터로 전달. 필요 없는 데이터까지 전달될 수 있음★ 25년 1회
제어 결합도제어 신호(플래그, switch 값)를 전달하여 다른 모듈의 내부 논리 흐름을 제어★★ 21년 3회, 24년 2회
외부 결합도외부에서 도입된 데이터 형식, 통신 프로토콜, 디바이스 인터페이스 등을 공유—
공통 결합도여러 모듈이 전역 변수를 함께 참조·수정★ 25년 1회
내용 결합도한 모듈이 다른 모듈 내부의 변수·기능에 직접 접근. 가장 나쁜 결합. 독립성 완전 파괴★ 25년 1회, 가장 강함
03
응집도 (Cohesion) — 높을수록 Good
20년 1회24년 1회24년 2회

모듈 내부의 기능적인 집중 정도. 하나의 모듈이 하나의 기능에 집중할수록 응집도가 높다.

응집도 스펙트럼
왼쪽(높음, 좋음) ──────────────── 오른쪽(낮음, 나쁨)
👍 높음(집중적) 👎 낮음(산만함)
기능
Functional
단 하나의 기능만 수행
순차
Sequential
이전 출력이 다음 입력으로 연결
교환/통신
Communication
동일 데이터를 다루는 기능 묶음
절차
Procedural
정해진 절차 순서대로 수행
시간
Temporal
특정 시점에 함께 수행 (초기화 등)
논리
Logical
논리적으로 비슷한 기능들 묶음
우연
Coincidental
아무 관계 없이 우연히 묶인 것
💡 암기법 — 응집도 높음→낮음

기순교절시논우 — 기능 / 순차 / 교환 / 절차 / 시간 / 논리 / 우연
"기순교절, 시논우~" — 기능이 가장 좋고 우연이 가장 나쁨!

24년 2회 기출 — 순차적 응집도 설명

"모듈 내부의 각 기능들이 서로 연관성을 가지고 순차적으로 수행되는 응집도. 하나의 작업이 끝나면 그 결과가 다음 작업의 입력으로 사용" → 순차적(Sequential) 응집도

24년 1회 기출 — 높은 순 배열 (보기: ①기능 ②교환 ③우연 ④시간)

정답: ① 기능 > ② 교환 > ④ 시간 > ③ 우연

05
Fan-in / Fan-out
20년 1회
📥 Fan-in
어떤 모듈을 호출하는 모듈의 수. 해당 모듈로 들어오는 화살표 수. 높을수록 재사용성↑이지만 과도하면 유지보수 복잡.
Fan-in = 나를 부르는 모듈 수
📤 Fan-out
어떤 모듈이 호출하는 모듈의 수. 해당 모듈에서 나가는 화살표 수. 높을수록 결합도↑, 복잡도 증가.
Fan-out = 내가 부르는 모듈 수
20년 1회 기출 — Fan-in 2 이상인 모듈 찾기

모듈 구조도에서 위에서 두 개 이상의 화살표가 들어오는 모듈을 찾는 문제. 여러 모듈에서 공통으로 호출되는 하위 모듈이 해당됨.

팁: Fan-in이 높은 모듈 = 공통으로 쓰이는 유틸리티 모듈 (좋은 설계)
Fan-out이 높은 모듈 = 많은 것을 제어하는 복잡한 모듈 (주의 필요)
