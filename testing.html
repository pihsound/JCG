01
블랙박스 vs 화이트박스
20년 3회22년 1회
⬛ 블랙박스 테스트
내부 구조를 모르는 상태에서 입력값과 출력값만으로 검증. 명세 기반 테스트.
  • 동치 분할 (Equivalence Partitioning)
  • 경계값 분석 (Boundary Value Analysis)
  • 원인-결과 그래핑 (Cause-Effect Graphing)
  • 오류 예측 (Error Guessing)
  • 비교 테스팅 (Comparison Testing)
  • 페어와이즈 테스팅
⬜ 화이트박스 테스트
내부 코드 구조와 로직을 알고 수행. 모든 경로를 검증. 구조 기반 테스트.
  • 기초 경로 검사 (Base Path Test)
  • 루프 검사 (Loop Test)
  • 데이터 흐름 검사 (Data Flow)
  • 조건 검사 (Condition Testing)
  • 구조 검사 (Structure Testing)
💡 암기법

블랙박스 = 냉장고 외부에서 온도만 확인 (내부 회로 모름)
화이트박스 = 냉장고 내부 회로를 직접 들여다보며 검사

02
블랙박스 테스트 기법
21년 1회21년 3회 22년 1회23년 3회
블랙박스
동치 분할 (동등 분할)Equivalence Partitioning
★★★★

입력 데이터를 동일한 결과를 내는 등가 클래스(구간)로 나누어, 각 클래스에서 대표값 하나만 테스트. 테스트 케이스 수를 최소화하면서 효율성 극대화.

예시 — 점수 입력 (0~100 유효)

유효 클래스: 0~100 → 대표값 50 하나만 테스트
무효 클래스1: -1 이하 → 대표값 -1
무효 클래스2: 101 이상 → 대표값 101

블랙박스
경계값 분석Boundary Value Analysis
★★★★

오류는 경계(끝 값)에서 가장 많이 발생한다는 원칙. 유효 범위의 경계 값과 그 바로 안팎의 값을 테스트.

예시 — 0 ≤ X ≤ 10 범위 (21년 1회 기출)

테스트 입력값: -1, 0, 1 (하한 경계) + 9, 10, 11 (상한 경계)

블랙박스
원인-결과 그래핑Cause-Effect Graphing
★★★

입력 데이터 간의 관계와 출력에 영향을 미치는 상황을 체계적으로 분석하여 효용성 높은 테스트 케이스 선정. 복잡한 조건 조합 테스트에 유용.

💡 기출 포인트

"입력 데이터 간의 관계와 출력에 영향을 미치는 상황을 체계적으로 분석하여 효용성 높은 검사 사례를 선정"
→ 원인-결과 그래핑(Cause-Effect Graphing)

블랙박스
오류 예측Error Guessing

테스터의 경험과 직관을 바탕으로 오류가 발생할 만한 입력값을 예측해 테스트. 공식적인 절차는 없으나 숙련된 테스터에게 효과적.

03
커버리지 (Coverage) — 최빈출!
21년 2회22년 1회 23년 2회24년 3회 25년 1회

테스트가 소스코드를 얼마나 실행했는지 측정하는 지표. 화이트박스 테스트에서 사용. 강도: 문장 < 분기 < 조건

①
문장 커버리지
Statement Coverage
모든 구문(명령문)이 최소 한 번 이상 실행되었는지 확인. 가장 약한 기준.
②
분기 커버리지
Branch / Decision Coverage
모든 분기(if·switch)의 True/False 결과가 각각 한 번 이상 실행. 문장 커버리지를 포함.
③
조건 커버리지
Condition Coverage
각 조건식이 True·False를 모두 한 번씩 가지도록 테스트. 가장 강한 기준.
코드로 보는 커버리지 차이 (24년 3회 유형)
int score = 85;
boolean pass = false;

if (score >= 60 && score <= 100) {     ← 조건식: score>=60  AND  score<=100
    pass = true;
}
System.out.println(pass);
커버리지필요한 테스트 케이스설명
문장score=85 (1개로 충분)모든 구문을 한 번만 실행하면 OK
분기score=85 (true), score=50 (false)if문의 참·거짓 경로 모두 실행
조건score>=60 T/F, score<=100 T/F 각각개별 조건식이 T/F를 모두 가져야

🎯 커버리지 판별 연습

설명을 읽고 어떤 커버리지인지 선택하세요

① "소프트웨어 테스트의 효율성 지표로, 각 조건식이 모두 참과 거짓인 경우를 최소 한 번씩 평가되었는지 측정" (23년 2회 기출)
② "코드 상의 모든 문장이 최소 한 번 이상 실행되는지를 검사" (24년 3회 기출)
③ "if나 switch 같은 분기문이 각각 정상적으로 수행되는지 확인" (24년 3회 기출)
④ "각 결정점에서 모든 가능한 결정 결과를 최소 한 번 이상 실행하도록 요구하며, 모든 조건식이 참과 거짓이 되도록" (24년 1회 기출)

💡 암기법 — 스텁 vs 드라이버

하향식 = 스텁(Stub) — 하향식은 위에서 아래로 내려오니, 아직 안 만들어진 하위 모듈을 임시로 채우는 스텁이 필요
상향식 = 드라이버(Driver) — 상향식은 아래서 위로 올라오니, 위에서 호출해줄 상위 모듈 역할의 드라이버가 필요
"아래가 없으면 스텁으로 막고, 위가 없으면 드라이버로 태워라"

05
테스트 오라클 (Test Oracle)
20년 4회

테스트 수행 후 결과가 올바른지 판단할 수 있는 기준 또는 메커니즘.

참(True) 오라클
True Oracle
모든 입력값에 대해 기대하는 결과를 제공. 완전한 판단 가능. 비용 高.
샘플링(Sampling) 오라클
Sampling Oracle
특정 몇 개의 입력값에 대해서만 결과 제공. 일반·업무용에서 사용.
휴리스틱(Heuristic) 오라클
Heuristic Oracle
나머지 값들은 경험·직관으로 판단. 샘플링 오라클의 확장.
일관성 검사 오라클
Consistent Oracle
앱 변경 전·후의 결과가 동일한지 비교 검사. 회귀 테스트에 활용.
