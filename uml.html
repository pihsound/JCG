Section 01
What is UML?
🔍
UML (Unified Modeling Language)
소프트웨어 시스템을 시각화·문서화하며, 시스템의 구조와 동작을 표준화된 방식으로 명세하는 모델링 언어.
복잡한 시스템을 다양한 다이어그램으로 쉽게 표현하고, 요구사항과 기능을 명확히 정의한다.
UML의 4가지 특징
👁️
가시화 언어
Visualization
다이어그램으로 시스템 구조·동작을 시각적으로 표현
📋
명세화 언어
Specification
표준화된 언어로 시스템 요구사항·기능 명세
🏗️
구축 언어
Construction
설계·구현에 도움을 주며 객체지향 개발에서 중요한 역할
📄
문서화 언어
Documentation
기능·구조·동작을 표준화된 형태로 문서화
Section 02
UML 구성요소
UML = 사물 + 관계 + 다이어그램
🧱
사물 (Things)
모델의 요소
🔗
관계 (Relationships)
요소 간의 연결
📊
다이어그램 (Diagrams)
시각적 표현
사물 (Things)
🏛️
구조사물
Structural Things
시스템의 개념적·물리적 요소
예) 클래스, 유스케이스, 컴포넌트
⚡
행동사물
Behavioral Things
시간·공간에 따라 변화하는 행위
예) 상호작용, 상태머신
📦
그룹사물
Grouping Things
요소들을 그룹으로 묶어 관리
예) 패키지
📝
주해사물
Annotation Things
부가적인 설명·제약 조건
예) 주석, 노트
🔌
인터페이스
Interface
클래스가 제공해야 할 오퍼레이션 세트 정의
스테레오타입: «interface»
💡
암기 TIP
구성요소 3가지는 사관다 = 사물 + 관계 + 다이어그램으로 외우자!
인터페이스는 스테레오타입 «interface»로 표시한다는 것도 기출 포인트!
Section 03
Diagram 종류
Structural
구조 다이어그램
6종
🗂️클래스Class
객체들의 타입 정의·관계 도식화. 시스템의 구조화. 상속·연관·의존성 표현. ★★★ 빈출
🔲객체Object
시스템 내 객체와 이들 간 관계를 특정 시점에 시각화.
⚙️컴포넌트Component
소프트웨어의 논리적 구성요소(컴포넌트)와 이들 간의 의존성 나타냄.
🖥️배치Deployment
시스템의 물리적 배치와 구성요소 간 관계. 하드웨어·네트워크·소프트웨어 구성요소 배치 시각화.
🔩복합체 구조Complex Structure
시스템 내 복합 구조와 클래스/컴포넌트 내부 구조 및 상호작용 표현.
📦패키지Package
시스템의 패키지와 클래스 간 그룹화 및 관계 시각화. 20년4회 23년3회 빈출
Behavioral
행위 다이어그램
7종
👤유스케이스Use Case
시스템의 기능과 사용자 간 상호작용 표현. 구성요소: Actor, Use Case, System
⏱️순차Sequence
객체 간 상호작용을 시간 순서에 따라 표현. 구성: 활성 객체, 메시지, 생명선, 제어 사각형. 메시지 유형: 동기·비동기·반환·자체
💬커뮤니케이션Communication
객체 간 상호작용과 통신 표현. 시간 흐름보다 객체 간 관계와 통신에 중점.
🔄상태State
객체 생명주기 동안의 상태 변화 표현. 외부·내부 이벤트에 대한 반응 나타냄.
🏃활동Activity
시스템의 프로세스·워크플로우 표현. 일반적으로 비즈니스 프로세스 모델링에 사용.
🔀상호작용Interaction
시스템 상호작용을 고수준 워크플로우로 표현. 활동 다이어그램 + 순차 다이어그램의 혼합 형태.
🕐타이밍Timing
객체 행동과 시간에 따른 상호작용. 실시간 시스템이나 복잡한 상호작용 표현에 유용.
🧠
암기 TIP — 구조 다이어그램 6종
클객컴 배복패 = 클래스 / 객체 / 컴포넌트 / 배치 / 복합체 구조 / 패키지
🧠
암기 TIP — 행위 다이어그램 7종
유순커 상활상타 = 유스케이스 / 순차 / 커뮤니케이션 / 상태 / 활동 / 상호작용 / 타이밍
Section 04
UML 관계 6종
①
연관 관계 Association
─────────→
두 클래스가 서로 관련이 있음. 가장 일반적인 관계. 실선으로 표현.
②
의존 관계 Dependency
- - - - - →
한 클래스가 다른 클래스를 사용하는 관계. 점선 화살표. 변경 시 영향을 받을 수 있음.
③
일반화 관계 Generalization
────────▷
IS-A 관계. 한 클래스가 다른 클래스를 포함하는 상위 개념. 빈 삼각형 화살표. 상속 관계.
④
실체화 관계 Realization
- - - - - ▷
인터페이스와 구현 클래스 간의 관계. 점선 + 빈 삼각형. «interface»를 구현하는 관계.
⑤
집약 관계 Aggregation
◇─────────
HAS-A 관계. 전체 객체가 사라져도 부분 객체는 존재. 빈 마름모. 독립적인 생명주기.
🍖 불고기가 없어도 간장·다시다·미원은 따로 존재 가능!
⑥
합성 관계 Composition
◆─────────
강한 HAS-A 관계. 전체 객체가 사라지면 부분 객체도 함께 소멸. 채워진 마름모. 의존적 생명주기.
🪑 책상이 사라지면 책상다리·상판·나사도 함께 소멸!
집약 vs 합성 핵심 비교
구분 집약 (Aggregation) 합성 (Composition)
기호 ◇ (빈 마름모) ◆ (채워진 마름모)
관계 약한 HAS-A 강한 HAS-A
생명주기 독립적 (전체가 사라져도 부분은 OK) 의존적 (전체가 사라지면 부분도 소멸)
예시 불고기 ◇ 간장 (간장은 독립 존재) 책상 ◆ 책상다리 (분리 불가)

기출 Quiz

Q1
UML의 구성요소 3가지를 쓰시오.
✅ 사물(Things), 관계(Relationships), 다이어그램(Diagram)
Q2
클래스들 사이의 전체 또는 부분 관계로, 전체 객체가 사라져도 부분 객체는 독립적으로 존재하는 관계는?
✅ 집약 관계 (Aggregation) — 빈 마름모 ◇
Q3
한 클래스가 다른 클래스를 포함하는 상위 개념일 때의 IS-A 관계를 무엇이라 하는가?
✅ 일반화 관계 (Generalization)
Q4
구조 다이어그램 중, 시스템의 패키지와 클래스 간 그룹화 및 이들 간의 관계를 시각화하는 다이어그램은?
✅ 패키지 다이어그램 (Package Diagram)
Q5
UML에서 클래스와 같은 모델 요소 또는 컴포넌트가 구현해야 하는 오퍼레이션 세트를 정의하는 모델 요소는?
✅ 인터페이스 (Interface) — 스테레오타입 «interface»로 표시
Q6
순차 다이어그램(Sequence Diagram)의 구성요소 4가지를 쓰시오.
✅ 활성 객체, 메시지, 생명선(Lifeline), 제어 사각형
Q7
다음 UML 관계를 쓰시오.

② 한 클래스가 다른 클래스를 포함하는 상위 개념일 때의 IS-A 관계
③ «interface»를 구현하는 관계
